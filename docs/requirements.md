## 1. プロジェクト概要

### 目的

ZMKファームウェアを使用する自作キーボードのキーマップ変更を、
**GitHubリポジトリ（ソース）** と **キーボード本体（フラッシュメモリ）** に対して
UIから一括・同期的に反映できるアプリケーションを開発する。

### 背景・課題

**課題①：既存ツールのトレードオフ**

| ツール | できること | できないこと |
|---|---|---|
| Keymap Editor | 複雑なビヘイビア設定 | 実機への即時反映 |
| ZMK Studio | 実機への即時反映 | 複雑なビヘイビア設定 |

**課題②：キーマップ情報の喪失**

ZMK Studioで手軽に変更したキーマップ情報を保持したまま、
GitHubのソースをいじってファームウェア全体に変更を加えることができない。
（再ビルド・再フラッシュするとStudioでの変更が消える）

### 解決アプローチ

UIからキーマップを変更した際に以下を同時実行し、**結果的同期**の状態を保つ：

- **即時バイナリ**：ZMK Studio RPCプロトコル経由で実機の **settingsパーティション** を直接書き換え → 実機に即時反映
- **ビルドバイナリ**：GitHubにソースをpush → GitHub Actionsでビルド → GitHubにソースとして記録

※ ビルドバイナリは実機に焼かない。GitHubのソースが実機と同じキーマップを持つことを保証する目的で使用する。

**【重要】ZMK RPCとsettingsパーティションについて**

ZMK Studio RPCによるキーマップ変更は、ファームウェア本体ではなく **settingsパーティション（NVS領域）** に保存される。
このため以下の性質を持つ：

- settingsパーティションに変更が存在する限り、再フラッシュ後もRPC変更済みの値が優先される
- ファームウェアを手動で再フラッシュしても、RPC変更は消えない（「Restore Stock Settings」実行時のみリセットされる）
- よって **GitHub側の変更を実機に反映する正式な経路は「RPC適用」のみ** であり、手動フラッシュで解決できない

これにより「手動フラッシュ後にGitHubのキーマップが実機に反映される」という従来想定は誤りであると確認された。
本アプリはRPC経由での差分適用機能（Task 2）によってこの問題を解決する。

## 2. 機能要件

### 2-1. キーマップ編集UI

- レイヤー・キーの視覚的編集（Keymap Editor相当）
- 編集内容はビルド実行まで一時保存状態として保持
- 対応する変更ファイル：`.keymap`（主）、`.conf`（ビヘイビア初回作成時）
- GitHub上で直接ソースが変更された場合も、アプリ起動時に実機から逆読み込みして表示し、GitHubのソースとズレがあれば警告を表示する

### 2-2. ビルド・同期処理（「保存」ボタン）

- `.keymap`（および必要時`.conf`）をGitHubリポジトリに書き換え・push
- GitHub ActionsによるZMKビルドを起動
- ZMK Studio RPCプロトコル経由でキーボード本体のフラッシュを直接書き換え（即時バイナリ）
- ビルド完了通知をUIに表示
- ビルド失敗時はエラー通知と手動リトライ案内を表示

### 2-3. ビルド中のUI制御

- ビルド実行中（queued / in_progress）は **保存ボタンのみ** 無効化する
- 保存ボタン無効化中は「ビルド中のため保存できません」のツールチップを表示する
- キーマップ編集（キー変更・レイヤー切替・コンボ編集）はビルド中も自由に行える
- 「実機に書き込み」ボタンはビルドと独立のためロックしない
- ビルド中に加えた編集は通常の未保存編集として扱い、ビルド完了後に保存できる
- ビルド完了（success / failure）で保存ボタンを再有効化する
- ビルドステータスをUI上（保存ボタン横）にバッジで表示する

### 2-4. GitHub連携

- Keymap Editorと同様のGitHubリポジトリ連携
- OAuth認証によるリポジトリアクセス
- `.keymap` / `.conf`の読み取り・書き込み

### 2-5. キーボード接続

- USB（WebSerial）またはBLE（WebBluetooth）経由で接続
- ZMK Studio RPCプロトコルを使用した通信
- 接続状態の表示

### 2-6. 実機からの逆読み込み

RPC経由で実機の現在のキーマップ状態を読み取り、UIに反映する。
以下のシナリオで使用する：

| シナリオ | 内容 |
|---|---|
| アプリ起動時 | 実機から現在のキーマップを読み込んでUIに表示（実機を正とする） |
| プッシュ失敗時 | 実機から読み直して改めてビルドプッシュ可能にする |
| GitHub直接編集後 | GitHub上でソースを直接編集した場合、実機には反映されていないため差分をRPCで適用する |

**【注意】手動フラッシュ後のシナリオについて**

settingsパーティションの仕様上、ビルドバイナリを手動で実機に焼いても、
RPC変更済みキーはsettingsパーティションの値が優先されるため実機のキーマップは変わらない。
手動フラッシュ後に実機をGitHubと一致させるには、本アプリからRPCで差分を適用する必要がある。
（または「Restore Stock Settings」でsettingsをリセット後に再フラッシュする方法もある）

### 2-7. ビヘイビア管理

**.confの扱い**

- `.conf`の読み込みを行い、各ビヘイビアの作成済み／未作成を判定する
- `.conf`への書き込みは専用画面からのみ行う（メイン画面からは変更不可）

**未作成ビヘイビアの表示**

- メイン画面の隅に未作成ビヘイビアの一覧を表示する
- 例：「以下のビヘイビアはまだ作成されていません。combo / tap-dance / macro …」
- 各ビヘイビアに解説ページへのリンク（？マーク）を設置する

**ビヘイビア専用作成画面**

- 未作成ビヘイビアを1つずつ作成する専用画面を用意する
- 作成時に`.conf`へ該当CONFIGを追記する
- 一度作成したビヘイビアは以降`.keymapのみ`の変更で追加可能になる

**ビヘイビア解説ページ**

初学者向けに各ビヘイビアの解説ページを用意する。含める内容：

- ビヘイビア名と概要
- 具体的な使用例（例：「AとSを同時押しでEscになる」）
- 設定できるパラメータの説明

## 3. 非機能要件

| 項目 | 内容 |
|---|---|
| アプリ形態 | Webアプリ |
| 対応ブラウザ | Chrome / Edge（PC・Android）、Bluefy等の専用ブラウザ（iOS） |
| iOSでの利用 | Bluefy等WebBluetooth対応ブラウザが必要。UI上に案内を表示する |
| ビルド基盤 | GitHub Actions（クラウドビルド） |
| RPC通信 | ZMK Studio RPCプロトコル（USB/BLE） |
| 対象ファームウェア | ZMK Studio対応済みキーボードのみ（`CONFIG_ZMK_STUDIO=y`が前提） |

## 4. 対応スコープ（初期版）

### 対応する
- `.keymap`ファイルのキー割り当て変更
- 既存ビヘイビア（hold-tap, mod-morph等）の設定変更
- `.conf`への機能追加（ビヘイビア初回作成時のみ・専用画面から）

### 対応しない（初期版）
- ボード定義・デバイスツリーの変更
- `.conf`の依存関係の自動解決

## 5. システム構成イメージ

```
[編集UI]
    ↓（保存ボタン）
    ├─→ [GitHub API] → .keymap / .conf 書き換え → push
    │       ↓
    │   [GitHub Actions] → ZMKビルド → ビルドバイナリ生成（実機には焼かない）
    │
    └─→ [ZMK Studio RPC] → USB/BLE → キーボード本体フラッシュ書き換え（即時バイナリ）

[実機] ←→ [ZMK Studio RPC] ←→ [編集UI]（起動時・障害時の逆読み込み）

※ 即時バイナリ：実機に即時反映
※ ビルドバイナリ：GitHubのソース正確性の担保（実機へのフラッシュは行わない）
※ 両者は結果的に同じキーマップ動作を持つ（結果的同期）
※ 実機を正として扱う設計
```

## 6. 残課題

### 🔴 技術検証が必要（着手前に確認必須）

| # | 課題 | 内容 |
|---|---|---|
| T-1 | ZMK Studio RPC の外部利用可否 | WebSerial/WebBluetooth経由でWebアプリからRPCが呼び出せるか検証。公式TSクライアント（`@zmkfirmware/zmk-studio-ts-client`）での実装可否を確認する |

### 🟢 今後の拡張候補（初期版スコープ外）

| # | 課題 | 内容 |
|---|---|---|
| E-1 | 複雑なビヘイビア対応 | tap-dance、macro等の詳細パラメータ設定 |
| E-2 | 複数キーボードの管理 | 複数リポジトリ・複数デバイスの切り替え対応 |

## 7. 次のアクション

1. **T-1の検証**：WebSerial/WebBluetooth経由でZMK Studio RPCが呼び出せるか、公式TSクライアントで確認する
2. T-1が可であればアプリの基本構成設計へ進む
3. T-1が否であればTauriまたはElectronによるデスクトップアプリへの切り替えを検討する
