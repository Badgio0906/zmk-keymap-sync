# ZMK Keymap Sync — 利用仕様書

更新日: 2026-07-06（Task R・S・T・U 完了後）

---

## 1. 概要

ZMK Keymap Sync は、ZMKファームウェアを使う自作キーボードのキーマップを
Webブラウザから変更するアプリ。単一 HTML ファイル (`app/index.html`) で動作し、
外部ライブラリは使用しない。

**変更を行うと以下を同時に実行する:**

1. **即時バイナリ反映**: WebSerial 経由の ZMK Studio RPC で実機の settingsパーティションを直接書き換え
2. **ソース同期**: GitHub に `.keymap` ファイルを push → GitHub Actions でビルド（ファームウェアは焼かない）

デバイス未接続の場合は GitHub push のみ実行し、接続後に「実機に書き込み」ボタンで差分を適用できる。

---

## 2. 前提条件

### 2-1. 対応ファームウェア

以下の両条件を満たす ZMK Studio 対応ファームウェアが必要。

```
1. CONFIG_ZMK_STUDIO=y でビルドされていること
2. studio-rpc-usb-uart スニペットが適用されていること
```

ファームウェア設定リポジトリの `build.yaml` を確認:

```yaml
- board: <board名>
  shield: <shield名>
  snippet: studio-rpc-usb-uart   # ← これが必須
```

シールドの `.conf` ファイルに `CONFIG_ZMK_STUDIO=y` が設定されていること。

**注意**: BLE（WebBluetooth）接続は未実装。USB のみ対応。

### 2-2. ブラウザ要件

- Chrome または Edge（PC・Android）
- iOS: Bluefy 等の WebSerial 対応ブラウザ
- `file://` では WebSerial が動作しないため、必ず HTTP サーバー経由でアクセスする

### 2-3. 起動方法

```powershell
npx serve app
# → http://localhost:3000/index.html
```

VS Code の Live Server 拡張でも可。

### 2-4. ZMK Studio との排他

同一シリアルポートは同時使用不可。ZMK Studio を開いている場合は閉じてから接続する。

## 4. 初期設定

### 4-1. PAT 設定

ヘッダー右の「⚙ PAT設定」をクリックし、GitHub Personal Access Token を入力する。
PAT は `localStorage` に保存される（ページリロード後も保持）。

必要なスコープ: `repo`（キーマップリポジトリへの読み書き）

### 4-2. キーボード追加

ヘッダー左のキーボード切替ドロップダウン →「＋ キーボードを追加」

設定項目:
- キーボード名（表示名）
- GitHub リポジトリ URL（貼り付けると owner/repo/branch を自動解析）
- `.keymap` ファイルパス
- レイアウト `.json` ファイルパス（KLE 形式）

「自動検出」ボタンでリポジトリ内の `.keymap` と `.json` を自動探索できる。

## 5. キーマップ読み込み

### 5-1. GitHub から読み込む

アクションバーの「↻ GitHub」ボタン、またはページロード後に自動実行される。
`.keymap` の内容をパースしてレイヤー・バインディングを取得する。

### 5-2. 実機から読み込む（実機接続時のみ）

アクションバーの「📥 実機」ボタン（USB 接続後に表示される）。
ZMK Studio RPC（GetKeymap）で実機の settingsパーティションを読み取り、
GitHub のキーマップと照合して差分を検出する。

差分がある場合は**差分パネル**が表示される（→ セクション 8）。

### 5-3. 表示中データバッジ

ステータスパネルの「表示中データ」に現在の表示元が表示される:

- `GitHub (main)` — GitHub から読み込んだデータを表示中
- `実機 (HH:MM:SS 読込)` — 実機から読み込んだデータを表示中

未保存の変更がある場合は件数も表示される。

## 6. キー編集

### 6-1. 基本操作

キーボードキャンバスのキーをクリックするとキー編集モーダルが開く。

左側でビヘイビアを選択し、右側にそのビヘイビアの説明・設定・「⚡ 実機書き込み対応」バッジが表示される。
プレビュー入力欄で直接テキスト編集も可能。

### 6-2. 対応ビヘイビア（標準）

| ビヘイビア | 説明 | 実機書き込み |
|---|---|---|
| `&kp` | キー入力（ビジュアルキーボードで選択） | ✓ |
| `&lt` | レイヤーホールド＋タップキー | ✓ |
| `&mt` | モディファイアホールド＋タップキー | ✓ |
| `&none` | 何もしない | ✓ |
| `&trans` | 下位レイヤーに透過 | ✓ |
| `&bootloader` | ブートローダーモード | ✓ |
| `&mkp` | マウスボタン（MB1〜MB5） | ✓ |
| `&bt` | Bluetooth操作（BT_SEL/NXT/PRV/CLR/CLR_ALL） | ✓ |

**注意**: `&mo`, `&to`, `&tog` にはボタンがないが、プレビュー入力欄から直接入力可能。

### 6-3. カスタムビヘイビア（tap-dance）

`.keymap` の `behaviors {}` ブロックに tap-dance が定義されている場合、
キー編集モーダルの下部「カスタムビヘイビア」セクションにボタンが表示される。

- 「**⏳ フラッシュ待ち**」バッジ付きのボタン: tap-dance は定義済みだが実機にファームウェアが焼かれていない状態。実機書き込みは不可。
- バッジなし（またはRPCバッジ）: 実機への即時書き込みが可能。

ボタンをクリックすると右パネルに tap-dance の詳細（タップ数・各動作・フラッシュ状態）が表示される。
「キーに割り当て」ボタンでキーボードをクリックして直接キーを選んで割り当てられる（→ セクション 7）。

## 7. ビヘイビア管理（tap-dance）

アクションバーの「🎵 ビヘイビア管理」ボタン（キーマップ読み込み後に有効化）。
USB 接続なしでも使用可能。

### 7-1. TD 一覧

定義済み tap-dance の一覧が表示される。各エントリに:
- 名前・タップ数・各動作の概要
- 「フラッシュ待ち」バッジ（実機に反映されていない TD に表示）
- 割り当て済みキー数

### 7-2. 新規 TD 作成

「＋ 新規作成」ボタン。設定項目:
- TD 名（`&` なしで入力）
- タップ時の動作（`&kp A` 等）、最大 5 つまで追加可能
- tapping-term-ms（デフォルト 200ms）
- キー位置: 数値直接入力 **または** 「キーボードから選ぶ」でキャンバスのキーをクリックして選択

「GitHubに保存」で `.keymap` の `behaviors {}` ブロックに DTS フォーマットで追記し、GitHub push する。

### 7-3. 既存 TD 編集

TD エントリの「編集」をクリック。作成フォームと同じ操作で変更・保存できる。
編集モード中はキャンバス上の割り当て済みキーが紫色でハイライトされる。

### 7-4. TD 削除

TD エントリの「削除」をクリック → 確認パネルが展開し、削除対象 TD が割り当てられているキー一覧が表示される。「削除実行」で:
1. `behaviors {}` ブロックから TD 定義を削除
2. 割り当てられていたキーをすべて `&none` に変更
3. GitHub push

### 7-5. フラッシュ待ち状態

新規 TD を作成・保存した後、実機にファームウェアを**手動フラッシュ**するまでは「フラッシュ待ち」状態になる。

フラッシュ待ち TD は:
- キー編集モーダルで「⏳ フラッシュ待ち」バッジ表示（実機書き込み不可）
- ステータスパネルに「フラッシュ待ち」注意書き表示
- 差分パネルで「適用不可（フラッシュ待ち）」と表示

フラッシュ待ち解除方法:
1. GitHub push（「💾 保存」）でビルド成功を確認
2. 生成されたファームウェアを実機に手動フラッシュ
3. 「📥 実機」で再読み込み → フラッシュ待ちバッジが消える

## 8. 差分パネル（実機 / GitHub 差分）

「📥 実機」で読み込んだ後、GitHub の値と実機の値が異なるキーがある場合に表示される。

差分テーブルの列:
- **レイヤー** / **キー位置**
- **GitHub の値**（保存済み）
- **実機の値**（現在動作中）
- **状態**（適用可 / 適用不可の理由）

### 8-1. GitHub → 実機（⬇ GitHubを実機に適用）

GitHub 側の値で実機を上書きする。

適用可能なキーのみ処理。適用不可の理由:
- `behavior ID 未確定` — `buildBehIdMap()` で behavior 名→ID の対応が学習できていない
- `フラッシュ待ち` — tap-dance が実機に未反映
- `未対応構文` — パース不可のバインディング

適用不可が残る場合は4ステップの案内を表示:
1. 未保存の変更をすべて GitHub に保存
2. GitHub ビルドでファームウェアを生成し手動フラッシュ
3. Restore Stock Settings で実機の RPC 変更をリセット
4. 「📥 実機」で再読み込みして一致を確認

### 8-2. 実機 → GitHub（⬆ 実機をGitHubに反映）

実機の値を `S.edits` に記録し、「💾 保存」で GitHub push できる状態にする。

## 9. 保存フロー

「💾 保存 (GitHub + 実機)」ボタン。

```
1. GitHub から最新 SHA を再取得（SHA 競合チェック）
   → 差異あり: SHA 競合モーダルを表示（リロードまたはキャンセル）

2. validateKeymap() で編集後キーマップの構文チェック
   → エラーあり: 保存をブロックしてエラー内容を表示
   → selfTestValidate() により raw + reconstruct 両方で 0 件を確認

3. reconstructKeymap() で .keymap テキストを再構築

4. GitHub API PUT でプッシュ

5. pollBuildStatus(commitSha) をバックグラウンド起動（GitHub Actions 監視）

6. デバイス接続中の場合:
   a. GetKeymap RPC で実機の behaviorId を取得
   b. buildBehIdMap() で behavior 名→ID マップを更新
   c. SetLayerBinding RPC で変更キーを書き込み
```

デバイス未接続時は GitHub push（手順 1〜5）のみ実行する。
接続後に「⚡ 実機に書き込み」ボタンで実機側の書き込みのみを実行できる。

### 保存バリデーション（N-3 再設計版）

`extractBlock()` で `keymap {}` ブロックのみを抽出してから、その中の `bindings = < >` ブロックを
ブロック単位で検証する。グローバルな `>;` カウントは使用しない（behaviors / combos の終端と混同するため）。

チェック内容:
- `bindings = < >` ブロックが閉じていること
- 各バインディングが `&` で始まること
- 変更後のバインディング数が変更前と一致すること（レイヤーごと）

## 10. GitHub Actions ビルド監視

保存後、バックグラウンドでビルド状態をポーリングする。

ステータスパネルの「ビルド」セクション（ビルド中のみ表示）:

| 表示 | 意味 |
|---|---|
| `⏳ ビルド開始待ち` | push 直後、run 検出前（10秒×10回） |
| `⏳ ビルド待機中` | run がキューに入っている |
| `⚙ ビルド中` | Actions が実行中 |
| `✓ ビルド成功` | 完了（緑） |
| `✗ ビルド失敗` | 失敗（赤） |

ビルド中は「💾 保存」ボタンがロックされる。「⚡ 実機に書き込み」はビルドと独立して使用可能。

ビルド失敗時は Actions ログ URL がログに表示される。自動リトライはない。
Actions ワークフローが設定されていないリポジトリでは run 未検出でロック自動解除される。

## 11. コンボ編集

アクションバーの「🔗 コンボ編集」ボタン（USB 接続中のみ有効）。

`.keymap` 内の `combos {}` セクションのコンボを追加・編集・削除できる。
「keymap に反映」で `S.keymapRaw` を更新し、「💾 保存」で GitHub push される。

**注意**: コンボは実機への即時書き込み（RPC）には非対応。GitHub push のみ。

## 13. 既知の制約

| 制約 | 内容 |
|---|---|
| `&mo`, `&to`, `&tog` | キー編集モーダルにボタンなし（プレビュー入力欄で直接入力は可能） |
| コンボの即時書き込み | GitHub push のみ対応（RPC 未対応） |
| tap-dance 以外のカスタムビヘイビア | `behaviors {}` に `&name` で参照できるが、詳細 UI はない |
| BLE 接続 | 未実装（USB のみ） |
| `behaviors` RPC サブシステム | `ListAllBehaviors` 等は未実装（behavior ID は位置照合で推論） |
| ページリロード後のビルド状態 | 復元されない |
| キーボードスケール | 手動変更不可（自動のみ） |

## 14. settingsパーティションについて（重要）

ZMK Studio RPC による変更は **settingsパーティション（NVS領域）** に保存される。

- ファームウェアを**手動フラッシュしても RPC 変更は消えない**（Restore Stock Settings 実行時のみリセット）
- 「ビルドしたファームウェアを焼けば実機が GitHub と一致する」という前提は**誤り**
- GitHub 側の変更を実機に反映する正式な経路は「RPC 適用」のみ
- tap-dance のような firmware 定義のビヘイビアは、ファームウェアへの焼き込み後に RPC で利用可能になる
