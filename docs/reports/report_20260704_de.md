**旧挙動**:

## 修正E: トークンシステムによるポーラー制御

### 設計

`S.buildPollToken: number` — 整数。インクリメントで **全実行中ポーラーを一括無効化** する。

```
KB-A保存 → pollBuildStatus() → token = ++S.buildPollToken (=1)
  ↓ (10秒後、ポーリング中)
KB-B切り替え → S.buildPollToken++ (=2) → setBuildLock(false) → checkAndResumeBuild()
  ↓ KB-A の _pollRunLoop が次のループに入る
  → token(1) !== S.buildPollToken(2) → return  ← KB-A のポーラーが即座に終了
  ↓ checkAndResumeBuild が KB-B の最新run を確認
  → 進行中なら setBuildLock(true) + _pollRunLoop 再開（token=2）
  → 完了/なしなら build-status を hidden にしてロック解除したまま
```

### 各シナリオの修正後挙動

**シナリオ1**: KB-B に切り替えると KB-A のポーラーはトークン不一致で次の awaitチェック時に `return` → バッジ汚染なし ✓

**シナリオ2**: KB-B 保存時に `pollBuildStatus` が `++token` して 新セッション開始。KB-A のポーラーは既に終了済み（シナリオ1の時点で停止）→ 2本走行なし ✓

**シナリオ3**: KB-A に戻ると `S.buildPollToken++` + `checkAndResumeBuild()` が走る。KB-A の最新 run を1回取得 → queued/in_progress ならポーリング再開 + バッジ表示 ✓

### 実装上の注意点

- `_pollRunLoop(runId, pollOwner, pollRepo, token)` は `S.kb.owner`/`S.kb.repo` を引数で受け取る。KB 切り替えが発生しても古い KB のリポジトリを正しく参照し続ける。
- `checkAndResumeBuild()` は `S.buildPollToken` を **インクリメントせず** 現在値を読む。KB 切り替え時に外側で既にインクリメントされているため、resume セッションが既存 token と共存できる。
- `S.pat` 未設定時は `checkAndResumeBuild()` が即座に return する（ghFetch が認証エラーになるため）。
