## G-1改：送受信16進ダンプ診断

### 実装内容

**送信hexダンプ（全RPC呼び出し）**

`serialRpc()` 内の `writer.write()` 直前に送信パケット全バイトのhex文字列をログ出力。

**送信エラーの明示（従来は握りつぶし可能性あり）**

`writer.write()` 例外発生時に `log-error` でエラーメッセージを出力してから rethrow。

**受信データhexダンプ（タイムアウト時のみ）**

受信バイトを `rawBytes[]` に蓄積し、タイムアウト時に以下を出力:
- 受信バイト数 = 0: 「受信データなし」メッセージ
- 受信バイト数 > 0（フレーム解釈失敗）: バイト数とhexダンプ（最大64バイト）

## G-3：書き込み結果の正確な表示

### 実装内容

`writeToDevice()` がすべての終了パスで `{ ok: true/false }` を返すよう変更:

| 終了パス | 戻り値 |
|---|---|
| GetKeymap タイムアウト | `{ ok: false }` |
| GetKeymap パース失敗（0バインディング） | `{ ok: false }` |
| 実機とGitHub一致（書き込み不要） | `{ ok: true }` |
| 差分書き込み完了 | `{ ok: true }` |

呼び出し側（保存ハンドラ・btn-write-deviceハンドラ）で戻り値を確認:
- `ok: true` → `✓ 実機書き込み完了`
- `ok: false` → `⚠ 実機書き込み未実施（キーマップ読み取り失敗）`

## 診断ログの読み方

| ログ内容 | 意味 | 次のアクション |
|---|---|---|
| `受信データなし` | デバイスがRPCに無応答 | ZMK Studio RPC (`CONFIG_ZMK_STUDIO=y`) が有効か確認 |
| `受信バイト数=N, フレーム解釈失敗` | データは来るが byte-stuffing フレームとして解釈できない | hexダンプを確認しプロトコル版差異を調査 |
| `送信エラー: ...` | WebSerial の write() が失敗 | USB接続が切れていないか・ポートが正常か確認 |
| `⚠ 実機書き込み未実施` | 上記いずれかで書き込みをスキップ | hexダンプを手がかりに原因を特定 |
