作成日: 2026-07-04

## 1. 不具合の調査結果

### 症状（ユーザー報告）

- 保存（GitHub + 実機）でGitHubは更新されるが実機が変わらない
- ログに「実機に書き込み中... → 書き込む変更がありません → 実機書き込み完了」と出る
- 「実機に書き込み」ボタン単体でも書き込まれない
- 実機書き込みがビルド完了後に実行されているように見える（ログのタイムスタンプより）

### 仮説A の検証

> GitHub push成功後に `S.edits` → `S.githubSaved` へ移行しており、その後の実機書き込みステップが移行後の空の `S.edits` を参照している

**結果: 部分的に正しい。ただし直接の原因ではない。**

- `writeToDevice()` の source 選択ロジックは `S.edits` が空のとき `S.githubSaved` にフォールバックする
- 保存フロー内では `S.githubSaved = { ...S.edits }` の直後に `writeToDevice()` が呼ばれるため、データは正しく渡る
- **ただし**、保存フロー完了後に `S.githubSaved = {}` でクリアされるため、その後に `btn-write-device` を押すと `S.edits` も `S.githubSaved` も空になる

### 仮説B の検証

> 修正A（ID学習の厳格化）により canApply が全滅し、結果として書き込み対象0件と表示されている

**結果: 通常は影響しない。**

- `buildBehIdMap()` は変更されていない他のキーからも behavior ID を学習できる
- `&kp` のような一般的な behavior は多くのキーで使われるため、変更キーが除外されても ID 学習は成功する
- 全キーを同一 behavior から同一 param に変えた極端なケースでのみ問題になりうるが、通常使用では発生しない

### 仮説C の検証

> 実機書き込みの対象抽出ロジックが「未保存編集」のみを見る設計になっており、push後・再読み込み後は構造的に常に空になる

**結果: 根本原因。**

コードパス追跡:
1. 保存フロー完了 → `S.edits = {}`, `S.githubSaved = {}` で両方クリア
2. `btn-write-device` 単体押下 → `writeToDevice()` 呼び出し
3. `source = S.edits.length > 0 ? S.edits : S.githubSaved` → 両方空 → `editEntries.length === 0`
4. `log('書き込む変更がありません', 'log-info')` → early return
5. `btn-write-device` ハンドラで `log('✓ 実機書き込み完了', 'log-ok')` は必ず実行（戻り値を見ていないため）

→「書き込む変更がありません」と「実機書き込み完了」が連続して表示される症状と一致。

また、ビルド完了後の「実機書き込み」に見える現象の説明:
- `setBuildLock(true)` は `btn-write-device` も無効化していた（修正F 前の実装）
- ビルド完了で `setBuildLock(false)` → `btn-write-device` 再有効化
- ユーザーがこのタイミングで押す → 上記の「空振り」が起きる
- タイムスタンプでビルド完了後に見えた理由はこれ

### 結論

**仮説C が根本原因**（仮説A はその具体的な症状として現れる）。

`writeToDevice()` が「未保存編集リスト」を参照する設計になっており、push後・再読み込み後は構造的に常に空になる。

## 2. 実施した修正

### Task F-1: writeToDevice() の差分アプローチへの書き換え

**修正内容:**

- `S.edits` / `S.githubSaved` への依存を完全撤廃
- GetKeymap で実機の現在バインディングを取得
- `S.layers[li].bindings[ki]`（現在の表示値・未保存編集含む）と実機の差分を計算
- 差分キーのみ SetLayerBinding で書き込む
- 差分ゼロ時のメッセージを「実機はGitHubと一致しています（書き込み不要）」に変更

**差分アプローチの特性:**

| タイミング | bindings の内容 | originalBindings の内容 | 差分計算の結果 |
|---|---|---|---|
| 保存フロー内（push前） | 新しい値（編集済み） | 古い値（未push） | 差分あり → 書き込む |
| btn-write-device単体（push済み・実機古い）| 新しい値 | 新しい値 | 差分あり → 書き込む |
| btn-write-device単体（実機=GitHub）| 新しい値 | 新しい値 | 差分なし → スキップ |

**S.githubSaved の完全削除:**

以下の全箇所から `S.githubSaved` を削除:
- S オブジェクト定義
- `updateChangeBadge()` の savedCount ブランチ
- 保存ハンドラ
- `btn-write-device` ハンドラ
- 実機読み込みハンドラ
- `applyGithubToDevice()`
- `loadData()`

### Task F-1: 保存フローの実行順序変更

**修正前の順序:**
1. SHA 競合チェック
2. GitHub push → originalBindings 更新
3. setBuildLock(true) + pollBuildStatus 開始
4. 実機書き込み（S.githubSaved 使用）

**修正後の順序:**
1. SHA 競合チェック
2. **実機書き込み（push 前・即時反映）** ← 移動
3. GitHub push → originalBindings 更新
4. setBuildLock(true) + pollBuildStatus 開始

push 前に実機書き込みすることで:
- `originalBindings` がまだ古い値 → 実機も古い値 → `buildBehIdMap` で全 behavior が学習できる（最良の条件）
- ビルド完了を待たずに実機が即時反映される

### 修正F: ビルドロック粒度変更

**修正前:**
```javascript
function setBuildLock(locked) {
  S.building = locked;
  ['btn-save', 'btn-write-device'].forEach(id => { ... el.disabled = locked; });
  canvas.classList.toggle('build-locked', locked);
}
```

**修正後:**
```javascript
function setBuildLock(locked) {
  S.building = locked;
  const btn = document.getElementById('btn-save');
  if (btn) {
    btn.disabled = locked;
    btn.title    = locked ? 'ビルド中のため保存できません' : '';
  }
}
```

- `btn-write-device` のロックを解除 → ビルド中も実機書き込み可能
- `keyboard-canvas` のロック（pointer-events: none + opacity）を解除 → ビルド中もキー編集可能
- 保存ボタンのみロック。ツールチップに「ビルド中のため保存できません」を表示
- ビルド中に加えた編集は未保存編集として `S.edits` に積まれ、ビルド完了後に保存できる

`docs/requirements.md` の「2-3. ビルド中のUI制御」を新仕様に合わせて更新済み。
