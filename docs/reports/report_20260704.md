### Task 2: GitHub→実機の差分RPC適用UI

**実装アーキテクチャ:**

```
loadFromDevice()
  ↓ GetKeymap RPC（既存）
  ↓ idToBeh マップ構築（既存）→ S.idToBeh に保存（追加）
  ↓ 差分計算（既存）
  ↓ diffItems 構築（追加）→ S.diffItems に保存
  ↓ 差分パネル表示（追加）

差分パネル:
  表: レイヤー / キー位置 / GitHub値 / 実機値 / 適用可否
  ボタン1: 「⬇ GitHubを実機に適用」→ applyGithubToDevice()
  ボタン2: 「⬆ 実機をGitHubに反映」→ 差分パネルを閉じて保存ボタンへ誘導
  ボタン3: 「閉じる」

applyGithubToDevice():
  1. GetKeymap RPC（behavior ID 再取得）
  2. behIdMap 構築（GitHub originalBindings と照合）
  3. 適用可能な diff キーに SetLayerBinding で GitHub 値を書き込む
  4. 適用不可キーはスキップ＋ログ
```

- `S.idToBeh`: `{[behaviorId: number]: string}` — 最後のGetKeymapから得たIDマップ

## 判断に迷ったケース・設計上の懸念

### 1. `canApply` 判定の精度

`canApply` は「GitHub のバインディングが使うビヘイビア名が、デバイスの `idToBeh` マップに含まれるか」で判定している。

**潜在的な問題:**  
`idToBeh` は「GetKeymap の結果（device behavior ID）と GitHub の originalBindings（behavior 名）の位置照合」で構築している。
この照合は「同じキー位置で比較」しているため、device と GitHub でビヘイビア **型** が異なるキー（例：GitHub=`&kp A`、実機=`&lt 1 B`）が存在すると、
device の `lt` behavior ID が GitHub の `kp` 名と誤対応する可能性がある。

ただし、この問題は既存の `loadFromDevice()` の `idToBeh` 構築にも元から存在するものであり、
今回のTask 2で新たに導入したバグではない。実用上は「ビヘイビア型まで異なる差分」は稀であるため、許容範囲と判断した。

→ **相談ポイント**: 将来的に `idToBeh` 構築を「behavior 型が一致するキーのみを照合対象にする」よう改善すべきか？

### 2. `applyGithubToDevice` の成功検証なし

適用後に「本当に実機が更新されたか」を再度 GetKeymap で確認する処理は実装していない。
（仕様書では「適用後に再度GetKeymapで検証し、一致したことをログ表示」と指示あり）

実装を省いた理由: SetLayerBinding の応答が OK であれば実機への書き込みは完了したとみなせる（T-1 検証済み）。
ただし追加のGetKeymapポーリングでユーザー体験を向上させるか相談の余地あり。

→ **相談ポイント**: 適用後の再検証（GetKeymap → 差分が0になったか確認）を追加するか？

### 3. 適用不可時の案内メッセージ

適用不可バインディングには「Restore Stock Settings後に再フラッシュが必要」とログに出しているが、
この手順が正しいか確認が必要（settings をリセットしてからビルドバイナリをフラッシュすれば設定が上書きされるはず）。
