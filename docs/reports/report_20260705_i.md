# 調査I 実装レポート：Studio BLEトランスポートの検証

作成日: 2026-07-05
対象: ZMK本体ソース（`zmkfirmware/zmk` リポジトリ、ブランチ `v0.3-branch`）
※ `zmk-config-roBa` の `config/west.yml` が現在このブランチを指しているため、
　実際にビルドされるファームウェアのソースとして `v0.3-branch` を調査対象とした。
※ **コード変更は行っていない（調査のみ）**。

---

## 背景

`zmk-config-roBa` の2026-06-13コミット `c1c52e3` は、BLE接続の安定性向上を
目的に `CONFIG_ZMK_STUDIO=n` へ変更していた（コミットメッセージ:
「BLE接続安定性向上のため3本同時接続を2本に削減」）。
その後 `roBa_R` は `CONFIG_ZMK_STUDIO=y` + `studio-rpc-usb-uart` スニペット
の構成に変更済みだが、この構成が6月コミットの意図（BLE接続の削減）を
本当に守れているかは未検証だった。

---

## 調査1：`CONFIG_ZMK_STUDIO=y` で BLE(GATT) 側 RPC トランスポートはデフォルト有効になるか

`app/src/studio/Kconfig`（ZMK本体, v0.3-branch）を確認した。

```kconfig
menuconfig ZMK_STUDIO
    bool "Studio Support"
    select ZMK_STUDIO_RPC if !ZMK_SPLIT || ZMK_SPLIT_ROLE_CENTRAL
    ...

menuconfig ZMK_STUDIO_RPC
    bool "Remote Procedure Calls (RPC)"
    ...

if ZMK_STUDIO_RPC
menu "Transports"

config ZMK_STUDIO_TRANSPORT_UART
    bool "Serial"
    select SERIAL
    select RING_BUFFER
    default y if $(dt_chosen_enabled,$(DT_CHOSEN_ZMK_STUDIO_RPC_UART))

config ZMK_STUDIO_TRANSPORT_BLE
    bool "BLE (GATT)"
    select RING_BUFFER
    select BT_USER_DATA_LEN_UPDATE
    depends on ZMK_BLE
    default y
endmenu
endif
```

**結論: なる。** `CONFIG_ZMK_STUDIO_TRANSPORT_BLE` は `ZMK_BLE`（roBaのようなBLEキーボードでは常時有効）が
有効な限り **無条件に `default y`** である。UARTトランスポート（`CONFIG_ZMK_STUDIO_TRANSPORT_UART`）が
有効かどうかとは無関係に、BLE(GATT)トランスポートは独立してデフォルト有効になる。
両トランスポートは排他ではなく、**同時に両方コンパイル・登録され得る**。

## 調査2：`studio-rpc-usb-uart` スニペットはBLE側を無効化するか

スニペット本体（`app/snippets/studio-rpc-usb-uart/`）を直接取得して確認した。

**`studio-rpc-usb-uart.conf`:**
```
CONFIG_ZMK_USB=y
CONFIG_USB_DEVICE_STACK=y
CONFIG_USB_CDC_ACM=y
CONFIG_SERIAL=y
CONFIG_UART_INTERRUPT_DRIVEN=y
CONFIG_UART_LINE_CTRL=y
```

**`studio-rpc-usb-uart.overlay`:**
```dts
/ {
    chosen {
        zmk,studio-rpc-uart = &snippet_studio_rpc_usb_uart;
    };
};

&zephyr_udc0 {
    snippet_studio_rpc_usb_uart: snippet_studio_rpc_usb_uart {
        compatible = "zephyr,cdc-acm-uart";
    };
};
```

**結論: 無効化しない。** このスニペットは USB CDC-ACM を `zmk,studio-rpc-uart` の
chosen ノードに割り当てることで **UARTトランスポートを有効化するだけ**であり、
`CONFIG_ZMK_STUDIO_TRANSPORT_BLE` を `n` にする設定は一切含まれていない。
つまり、このスニペットを適用しても BLE(GATT) トランスポートは
（調査1の `default y` により）引き続き有効なままである。

参考として `app/src/studio/rpc.c` の `refresh_selected_transport()` を確認すると、
実際に「使用する」トランスポートは ZMK の現在の出力エンドポイント選択
（USB/BLEどちらへHID出力しているか）に応じて動的に切り替わる仕組みだった。
ただしこれは**実行時にどちらを使うかの選択**であり、BLE(GATT) RPCサービス
自体がビルド時に無効化されるわけではない。`CONFIG_ZMK_STUDIO_TRANSPORT_BLE=y`
である限り、GATTサービスはコンパイルされ実機上に存在し続ける。

## 調査3：BLE側トランスポートを無効化するKconfigシンボル

```
CONFIG_ZMK_STUDIO_TRANSPORT_BLE=n
```

roBa_R.conf への追加案（**適用は保留・未実施**）:

```diff
 CONFIG_ZMK_STUDIO=y
 CONFIG_ZMK_STUDIO_LOCKING=n
+CONFIG_ZMK_STUDIO_TRANSPORT_BLE=n
```

これにより BLE(GATT) 側の Studio RPC サービスをビルドから除外し、
USB(UART) トランスポートのみで ZMK Studio を利用する構成にできる
（6月コミットが意図していた「BLE接続を増やさない」状態に近づく）。

---

## 結論：6月コミットの意図は現構成で守られているか

**守られていない。**

現在の `roBa_R`（`CONFIG_ZMK_STUDIO=y` + `studio-rpc-usb-uart` スニペット）は、
USBトランスポートを追加で有効にしただけであり、**BLE(GATT)トランスポート
（`CONFIG_ZMK_STUDIO_TRANSPORT_BLE`）は明示的に無効化されない限りデフォルトで
有効**であることが判明した。6月コミット `c1c52e3` が `CONFIG_ZMK_STUDIO=n` に
していた本来の目的（BLE接続の増加を避け安定性を確保する）は、単に
`CONFIG_ZMK_STUDIO=y` + USBスニペットへ変更しただけでは実現されておらず、
BLE側のGATTサービスは以前と同様に有効化されたままになっている。

なお `roBa_R_coropit` も同一構成（`CONFIG_ZMK_STUDIO=y` + スニペットのみ、
`CONFIG_ZMK_STUDIO_TRANSPORT_BLE` の明示指定なし）であるため、
**同じ問題は roBa_R_coropit 側にも当てはまる**と考えられる
（coropit で実際にBLE不安定化が報告されているかは別途確認が必要）。

真に6月コミットの意図（BLE接続を増やさずStudioを使う）を満たすには、
調査3の `CONFIG_ZMK_STUDIO_TRANSPORT_BLE=n` を明示的に追加する必要がある。
この変更の適用可否は本レポート提出後にユーザーの判断を仰ぐ
（指示に基づき、今回は追加を保留し提案のみに留めた）。
