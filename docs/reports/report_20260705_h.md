# 修正H 実装レポート：キーコード対応の拡充と値レベル照合の徹底

作成日: 2026-07-05
対象: `app/index.html`

---

## 背景

実機/GitHub 差分一覧に表示された35件のうち大半が偽陽性だった。
原因は `parseZmkKeycode`（および内部テーブル `HID_KP`）が ZMK 公式の
キーコード別名（エイリアス）を網羅していなかったこと。GitHub 側の
`.keymap` 文字列を値へデコードできない場合、実機値との比較ができず、
実際には同一の値であっても「差分」として誤表示されていた。

## 調査（ZMK公式ドキュメント・ヘッダーを正として確認）

`zmk.dev` のキーコード一覧ページおよび ZMK 本体
`app/include/dt-bindings/zmk/keys.h` を直接取得し、以下を確認した。

- GUI系モディファイアは `LEFT_GUI`/`LGUI`/`LEFT_WIN`/`LWIN`/`LEFT_COMMAND`/
  `LCMD`/`LEFT_META`/`LMETA`（右側も同様）が同一コード（`0xE3`/`0xE7`）の別名
- テンキー番号は `KP_NUMBER_1`〜`9`/`KP_N1`〜`9`、`KP_NUMBER_0`/`KP_N0` が
  同一コード（`0x59`〜`0x62`）の別名。`KP_NUMBER_7` = `0x5F`（95）を実機側の
  実測例と突合し一致を確認
- `PLUS`・`EXCLAMATION` 等のシフト付き記号名は、ZMK ヘッダー上では
  `LS(EQUAL)`・`LS(NUMBER_1)` 等の**マクロ展開**であり、`.keymap` ソース上は
  フラットな1トークンとして書かれる。既存パーサは明示的な `LS(...)` 構文の
  ネストしか解けず、このマクロ展開後の別名トークンには対応していなかった

## 実装内容

### 1. `HID_KP` テーブルの拡充
- GUI/Win/Command/Meta の全別名（`LEFT_WIN`/`RIGHT_WIN`/`LEFT_COMMAND`/
  `LMETA` 等）を追加
- テンキー系（`KP_NUMBER_0`〜`9`、`KP_N0`〜`9`、`KP_PLUS`/`KP_MINUS`/
  `KP_MULTIPLY`/`KP_DIVIDE`/`KP_ENTER`/`KP_DOT`/`KP_EQUAL`/`KP_NUM`/
  `KP_COMMA` 等）を追加
- Non-US キー（`NUHS`/`NUBS`）を追加

### 2. `SHIFT_SYMBOL_ALIASES` テーブルを新設
シフト付き記号名（`PLUS`/`CARET`/`AMPERSAND`/`TILDE`/`UNDERSCORE`/
`EXCLAMATION`/`AT_SIGN`/`HASH`/`DOLLAR`/`PERCENT`/`PIPE`/括弧類等、
短縮別名含む）→ ベースとなる HID コードの対応表を追加。
`parseZmkKeycode` はこのテーブルにヒットした場合、LSビット（`0x02`）を
`modMask` に付与してエンコードする（ZMK の `LS(...)` マクロ展開と等価）。

### 3. `deviceBindingToString`（逆変換）に `SHIFT_SYMBOL_REVERSE` を追加
実機の値がシフト付き記号コード（modBits が LS 単独）に一致する場合は
対応するシフト記号名（例: `PLUS`）へ優先的に逆変換する。該当がなければ
従来どおり `LS(キー名)` 形式、それも解決できなければ16進表示を維持する
（要求仕様どおり「逆変換表に無いコードは16進表示を維持」）。

### 4. 差分判定の値レベル照合（既存機構の確認）
`loadFromDevice` は元々 GitHub 文字列をパースして実機の
`behaviorId`/`param1`/`param2` と**値レベルで**比較し、一致すれば
GitHub 表記をそのまま保持する設計になっていた。今回の対応で
`parseZmkKeycode` のカバレッジが広がったことにより、この既存の値比較
機構が正しく機能するようになった（構造自体の変更は不要だった）。

### 5. 「解析不能（アプリ側未対応）」の別枠表示を新設
GitHub 側の `.keymap` 構文が `parseBindingForRpc` でパースできない場合
（例: `&td`（tap-dance）等の未対応ビヘイビア）、**実機値と比較できない
以上「差分」として断定すべきではない**と判断し、以下のように変更した。

- 該当キーは `newBindings`（表示用配列）を実機値で**上書きしない**
  （従来は不明ビヘイビアでも上書きを試み、パース不能な構文を実機由来の
  推測文字列で潰してしまう恐れがあった）
- `S.diffItems`（差分件数・適用対象）には含めず、新設の
  `S.unparsableItems` に分離
- UI にも「解析不能（アプリ側未対応の構文）」という独立したセクションを
  追加（`diff-panel` 内、差分テーブルの下）。差分件数バッジとは別に件数を
  表示し、GitHub値のみを一覧表示する（実機への適用対象からは除外）

### 6. 網羅方針
ZMK docs 記載の標準セット（英語 US 配列の記号・GUI/テンキー系）を優先し、
tap-dance・macro 等の独自/複雑ビヘイビアは従来どおり
「解析不能」フォールバック（今回新設の別枠）で表示する方針とした。

## 修正後、例の35件がどう分類される見込みか

ユーザー報告にあった代表例（`LEFT_WIN` vs `LGUI`、`KP_NUMBER_7` vs `0x5f`、
`PLUS` vs `EQUAL` 等）はいずれも今回追加したエイリアス/シフト記号テーブルで
値が一致するようになるため、**「差分」から解消され非表示になる**見込み。

35件の内訳が「大半が偽陽性」という報告内容から推測すると:

- **解消される見込み（値が一致するようになる）**: GUI/Win/Command/Meta
  エイリアス絡み、テンキー番号エイリアス絡み、シフト付き記号名絡みの
  項目 — 報告例3種はすべてこの分類に該当し、おそらく大半を占める
- **「解析不能」に振り分けられる見込み**: tap-dance・macro等、独自
  ビヘイビアを使っているキー（もしあれば）。従来は「差分」として
  カウントされていたが、今回から独立枠に移動し、差分件数からは除外される
- **真の差分として残る見込み**: 上記いずれにも該当しない、実際に
  GitHub と実機の値が異なるキー（もしあれば）

実機での動作確認はユーザーによる実施が必要（ハードウェア接続が要る
WebSerial 経由の RPC 通信のため、本セッションでは実施不可）。
Node.js サンドボックスでの関数単体テストでは、上記3例すべてで値の
完全一致・逆変換の正常動作を確認済み（本レポート末尾の検証結果参照）。

## 動作確認（Node.js サンドボックスでの単体テスト）

```
LEFT_WIN vs LGUI          => 458979 / 458979  MATCH
KP_NUMBER_7 vs KP_N7      => 458847 / 458847  MATCH
KP_NUMBER_7 raw code      => 95 (0x5f)        一致確認
PLUS vs LS(EQUAL)         => 459310 / 459310  MATCH
EXCLAMATION vs LS(NUMBER_1) => 459294 / 459294 MATCH
deviceBindingToString(EQUAL code + LSビット) => "&kp PLUS"（逆変換確認）
parseBindingForRpc('&td 0 1')（未対応構文） => null（解析不能バケツへ）
```

スクリプト全体の構文チェック（`new Function()`）も実施し、エラーなし。
実機での動作確認（WebSerial 経由）はユーザー側での実施が必要。
