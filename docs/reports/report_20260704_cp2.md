## 各タスクの詳細

### Task 3: SHA競合対策 + beforeunload

**SHA競合対策:**
- 保存フロー内の `ghPutFile` 呼び出し直前に `ghGetFile` で最新 SHA を再取得
- 差異があれば保存を中止し、ユーザーにリロードを促すログを3行表示
- `S.edits` は保持（編集が失われない）
- 自動マージ・自動リベースは一切行わない

**beforeunload:**
- `window.addEventListener('beforeunload', ...)` で `S.edits` が空でない間は離脱確認ダイアログを表示
- `e.returnValue = ''` で現代ブラウザの仕様（カスタムメッセージ非表示）に準拠

**既存機能への影響:**
- `ghGetFile` の呼び出しが1回増えるが、競合がない通常フローでは0.1〜0.5秒の追加レイテンシのみ
- `finally` ブロックに `btn.textContent = '💾 保存 (GitHub + 実機)'` を追加（Task 4対応でテキストが変わる可能性への保険）

### Task 4: GitHub Actionsビルドステータス表示 + ビルド中ロック

**実装アーキテクチャ:**

```
ghPutFile 成功
  ↓
setBuildLock(true)        — btn-save/btn-write-device を disabled、keyboard-canvas を pointer-events:none
pollBuildStatus(commitSha) — バックグラウンドで走る（awaited しない）
  ↓
① run 検索フェーズ（最大100秒）
   GET /actions/runs?head_sha={sha}  を10秒間隔で最大10回
   → 見つかれば runId を取得してポーリングへ
   → 見つからなければ「ビルド設定なし」でロック解除
  ↓
② ステータスポーリング（20秒間隔、最大15分）
   GET /actions/runs/{runId}
   → queued → 待機中バッジ
   → in_progress → ビルド中バッジ
   → completed/success → ✓ ビルド成功バッジ、ロック解除
   → completed/failure → ✗ ビルド失敗バッジ＋Actionsログ URL 表示、ロック解除
   → 15分経過 → タイムアウトバッジ、ロック解除
```

**ビルドステータスバッジ（UI）:**
- action bar の `btn-save` の直後に `<span id="build-status">` を配置
- 状態別CSS: `bs-queued`（グレー）/`bs-progress`（ブルー）/`bs-success`（グリーン）/`bs-failure`（レッド）

**ロック対象:**
- `btn-save`: 保存ボタン（2重push防止）
- `btn-write-device`: 実機書き込みボタン
- `#keyboard-canvas`: `pointer-events: none + opacity: 0.65` でキー編集を視覚的に無効化

**ロックが解除されない状況の防止:**
- `pollBuildStatus().catch()` でエラー時も必ず `setBuildLock(false)` を呼ぶ
- KB切り替え・KB設定保存時にも `setBuildLock(false)` + バッジ非表示でリセット

## 設計上の懸念・判断に迷ったケース

### 1. SHA競合検出後の「リロード」導線

現在の実装はログメッセージのみ（「↻ GitHubボタンで再読み込みしてください」的な案内）。
UI上で専用の「再読み込みして編集を破棄」ボタンを出す方が UX 的には親切だが、
`next_tasks.md` では「再読み込み導線を提示（自動マージはしない）」とあり、ログで十分と判断した。

→ **相談ポイント**: 競合検出時に専用バナーまたはモーダルでリロードボタンを出すか？

### 2. ビルドロック中の btn-reload（GitHub再読み込み）の扱い

現実装ではビルド中でも「↻ GitHub」ボタンは有効のまま。
ユーザーがビルド中に GitHub を再読み込みすると `S.keymapSha` が更新され、
ポーリング中の `S.kb` が変わる可能性がある。

影響範囲:
- `pollBuildStatus` は `headSha`（commitShaのクロージャ）を持っているため、再読み込みしても監視は継続する
- 再読み込み後に GitHub の最新ファイルで表示が更新されるのは自然な動作

→ 特に問題なしと判断しロックしなかった。

### 3. レート制限（5,000 req/h）

run 検索フェーズ: 最大10回 = 10 req  
ポーリングフェーズ: 最大45回（15分 ÷ 20秒）= 45 req  
合計最大55 req／ビルド1回

PAT 認証時 5,000 req/h のレート制限に対して 55 req は無視できる範囲。

### 4. `putResult.commit.sha` の保証

`ghPutFile` が返す `putResult.commit.sha` は GitHub Contents API の仕様上常に存在する。
ただし `ghFetch` は `.json()` を返しており、API がエラーを返した場合は `ghFetch` 自体が throw するため、
`putResult.commit` が undefined になるケースは実質ない。
